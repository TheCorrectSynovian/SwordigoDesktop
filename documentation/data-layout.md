# Data Layout — Swordigo Desktop Runtime

> **Scope**: This document describes where Swordigo Desktop stores its runtime
> data, how paths are resolved at startup, and how the first-run setup copies
> data from a system install to the user's home directory.
>
> **Source of truth**: [`src/platform/data_path.h`](file:///home/quantumcreeper/SwordigoDesktop/src/platform/data_path.h),
> [`src/platform/data_path.cpp`](file:///home/quantumcreeper/SwordigoDesktop/src/platform/data_path.cpp)

---

## 1. Runtime Data Directory

All game data lives under a single **user-owned** directory, following the
"Minecraft-style" approach: the launcher and game always read from
`~/.local/share/swordigo-desktop/`. Users have full access to modify, mod, or
back up their data.

```
~/.local/share/swordigo-desktop/
├── assets/
│   └── resources/           Game textures (.pvr), models (.pod),
│                            scenes (.scene), scripts (.lua)
│
├── rl_assets/
│   └── resources/           RLSwordigo mod assets — replacement textures,
│                            models, and scripts for the RLSwordigo fork
│
├── engine/                  ARM binaries organised by version
│   ├── manifest.json        Pre-built binary registry (shipped with packages)
│   ├── v1.4.6/              Vanilla Swordigo v1.4.6
│   │   ├── armeabi-v7a/
│   │   │   └── libswordigo.so
│   │   └── arm64-v8a/
│   │       ├── libswordigo.so
│   │       └── libsre.so    (SRE engine, if compatible)
│   ├── v1.4.12/             Vanilla Swordigo v1.4.12 (primary target)
│   │   ├── armeabi-v7a/
│   │   │   └── libswordigo.so
│   │   └── arm64-v8a/
│   │       ├── libswordigo.so
│   │       └── libsre.so
│   ├── rl-v6.1/             RLSwordigo v6.1
│   │   └── armeabi-v7a/
│   │       ├── libswordigo.so
│   │       ├── libmini.so   (SwMini modloader)
│   │       └── libGlossHook.so
│   ├── sw3d/                3D camera mod
│   └── custom-*/            User-added custom instances
│       └── <arch>/
│           └── libswordigo.so
│
├── res/
│   └── raw/                 Music files (.mp3) — game soundtrack
│
├── save/
│   └── Documents/           Player save files (.gplayer format)
│                            NEVER packaged — user data only
│
├── cache/                   Temporary runtime cache (texture decode, etc.)
│
├── mods/                    User mods directory (future expansion)
│
└── src/
    └── assets/              Launcher textures (UI icons, backgrounds)
```

### Directory Details

| Directory | Contents | Writable | Notes |
|-----------|----------|----------|-------|
| `assets/resources/` | Game textures (`.pvr`), 3D models (`.pod`), scene definitions (`.scene`), Lua game scripts (`.lua`) | Yes | Vanilla Swordigo assets. Modders can replace files here directly. |
| `rl_assets/resources/` | Modified textures, models, and scripts for the RLSwordigo fork | Yes | Selected automatically when an `rl-*` engine binary is loaded. |
| `engine/` | ARM shared libraries per game version and architecture | Yes | See [Binary Selector](binary-selector.md) for instance management. |
| `engine/manifest.json` | Pre-computed binary registry | Yes | Generated by `BinarySelector::generate_manifest()` during packaging. Avoids filesystem scanning at boot. |
| `res/raw/` | Music `.mp3` files | Yes | Loaded by the SRE music system via host-side OpenAL playback. |
| `save/Documents/` | `.gplayer` save files | Yes | Created by the game engine. **Never bundled in packages.** |
| `cache/` | Runtime temporary files | Yes | Texture decode cache, decompressed assets. Safe to delete. |
| `mods/` | User mod directory | Yes | Reserved for the mod API. Not yet active. |
| `src/assets/` | Launcher UI textures | Yes | Boot screen backgrounds, button icons. |

## 2. Engine Directory Naming Convention

The `engine/` directory uses a naming convention to classify binaries:

| Prefix | Game Type | Assets Dir | Example |
|--------|-----------|------------|---------|
| `v*` | Vanilla Swordigo | `assets` | `v1.4.6`, `v1.4.12` |
| `rl-*` | RLSwordigo (mod) | `rl_assets` | `rl-v6.1`, `rl-v6.6` |
| `sw3d` | 3D Camera Mod | `assets` | `sw3d` |
| `custom-*` | User-added | Configurable | `custom-MyMod` |

Within each version directory, binaries are organised by CPU architecture:

```
v1.4.12/
├── armeabi-v7a/     ARM32 (32-bit)
│   └── libswordigo.so
└── arm64-v8a/       ARM64 (64-bit) — primary target
    ├── libswordigo.so
    └── libsre.so    (Swordigo Runtime Engine)
```

Mod directories (anything not matching `v*`) may contain companion `.so` files
alongside `libswordigo.so` — these are automatically detected as dependencies:

```
rl-v6.1/armeabi-v7a/
├── libswordigo.so       Main game binary
├── libmini.so           SwMini modloader
└── libGlossHook.so      Inline hooking library
```

## 3. Path Resolution Priority

When the runtime needs to locate a file (e.g., `engine/v1.4.12/arm64-v8a/libswordigo.so`),
it searches through **six locations** in priority order. The first path where the
file exists wins.

| Priority | Source | Path | Notes |
|----------|--------|------|-------|
| **1** | Environment variable | `$SWORDIGO_DATA_DIR/<relative>` | Override for testing. **Always used if set** (no existence check). |
| **2** | Compile-time define | `SWORDIGO_DATA_DIR_PATH/<relative>` | Set via `-DSWORDIGO_DATA_DIR_PATH=...` in Makefile. Checked for existence. |
| **3** | XDG user data | `$XDG_DATA_HOME/swordigo-desktop/<relative>` or `~/.local/share/swordigo-desktop/<relative>` | Primary runtime location. Created by first-run setup. |
| **4** | Current working directory | `./<relative>` | Development mode — run from project root. |
| **5** | Binary-relative | `<exe_dir>/<relative>` or `<exe_dir>/../<relative>` | Resolves via `/proc/self/exe`. Handles `/usr/bin/swordigo_boot` → `/usr/share/swordigo/`. |
| **6** | System install | `/usr/share/swordigo/<relative>` or `/usr/local/share/swordigo/<relative>` | Read-only fallback. RPM/DEB packages install here. |

### Resolution Logic

```
get_data_path("engine/v1.4.12/arm64-v8a/libswordigo.so")

1. $SWORDIGO_DATA_DIR set? → return $SWORDIGO_DATA_DIR/engine/v1.4.12/...
2. SWORDIGO_DATA_DIR_PATH defined at compile time?
   → check existence → if found, return it
3. ~/.local/share/swordigo-desktop/engine/v1.4.12/... exists?
   → return it
4. ./engine/v1.4.12/... exists?
   → return it  (developer running from project root)
5. readlink(/proc/self/exe) → parent dir → check relative path
   → return it
6. /usr/share/swordigo/engine/v1.4.12/... exists?
   → return it  (system package fallback)

Fallback: return "./engine/v1.4.12/..."  (will fail at load time)
```

> [!IMPORTANT]
> Priority 1 (`$SWORDIGO_DATA_DIR`) is returned **unconditionally** — the file
> is not checked for existence. This is intentional: the env var is an explicit
> override for testing and CI, where the caller knows the path is valid.

## 4. First-Run Setup

On first launch after a system install (RPM/DEB), the runtime detects that the
user data directory is empty and copies files from the system install location.

### Trigger Conditions

First-run setup runs when **all** of these are true:

1. `~/.local/share/swordigo-desktop/engine/manifest.json` does **not** exist
2. `~/.local/share/swordigo-desktop/assets/` does **not** exist
3. A system install exists at `/usr/share/swordigo/` (or `/usr/local/share/swordigo/`)

If no system install is found, the runtime checks for a CWD-based layout
(`./engine/manifest.json` + `./assets/`). If found, it operates in **development
mode** using the current directory directly — no copy is performed.

### What Gets Copied

| Source | Destination | Contents |
|--------|-------------|----------|
| `/usr/share/swordigo/engine/` | `~/.local/share/swordigo-desktop/engine/` | ARM binaries + manifest.json |
| `/usr/share/swordigo/assets/` | `~/.local/share/swordigo-desktop/assets/` | Game textures, models, scripts |
| `/usr/share/swordigo/res/` | `~/.local/share/swordigo-desktop/res/` | Music files |
| `/usr/share/swordigo/src/assets/` | `~/.local/share/swordigo-desktop/src/assets/` | Launcher textures |

Additionally, empty directories are created:
- `~/.local/share/swordigo-desktop/save/`
- `~/.local/share/swordigo-desktop/cache/`

### Implementation

```cpp
// data_path.cpp — ensure_user_data()
bool ensure_user_data() {
    // 1. Check if user data already exists
    if (exists(user_dir + "engine/manifest.json") && exists(user_dir + "assets/"))
        return false;  // Already set up

    // 2. Check for dev mode (CWD has data)
    if (sys_dir.empty() && exists("./engine/manifest.json"))
        return false;  // Dev mode

    // 3. Copy from system install
    const char* dirs[] = { "engine", "assets", "res", "src/assets" };
    for (auto dir : dirs)
        fs::copy(sys_dir + dir, user_dir + dir, recursive | overwrite);

    fs::create_directories(user_dir + "save");
    fs::create_directories(user_dir + "cache");
    return true;
}
```

## 5. Configuration Files

### User Instances — `~/.config/swordigo-desktop/instances.json`

Stores user-added custom binary instances. Only `custom-*` entries are written
here. System binaries come from `engine/manifest.json`.

```json
{
  "instances": [
    {
      "name": "[Custom] MyMod [ARM64] (Unknown)",
      "filename": "libswordigo.so",
      "filepath": "engine/custom-MyMod/arm64-v8a/libswordigo.so",
      "sha256": "abc123...",
      "version": "MyMod",
      "version_dir": "custom-MyMod",
      "arch": "ARM64",
      "file_size": 8523456,
      "is_default": false,
      "status": "unknown",
      "game_type": "Swordigo",
      "assets_dir": "assets",
      "icon_path": "",
      "dependencies": [],
      "dep_paths": []
    }
  ]
}
```

### Legacy Registry — `swordigo_binaries.json`

The old flat registry format, kept for backward compatibility. Contains a
`default` field and a `known_binaries` map keyed by SHA256 hash. Superseded by
`manifest.json` + `instances.json`.

```json
{
  "default": "engine/v1.4.12/armeabi-v7a/libswordigo.so",
  "known_binaries": {
    "<sha256>": {
      "filepath": "...",
      "version": "1.4.12",
      "arch": "ARM32",
      "game_type": "Swordigo",
      "status": "tested",
      "label": "v1.4.12 [ARM32] (Stable)"
    }
  }
}
```

### Controls — `<save_dir>/controls.ini`

Keyboard/gamepad button mapping configuration. Loaded by `InputConfig`.

## 6. System Install Layout

When installed via RPM or DEB package:

```
/usr/share/swordigo/
├── assets/resources/     Game assets (read-only)
├── engine/               ARM binaries + manifest.json (read-only)
├── res/raw/              Music (read-only)
└── src/assets/           Launcher textures (read-only)

/usr/bin/
└── swordigo_boot         Main executable

/usr/lib64/ (or /usr/lib/)
└── libsre.so             SRE engine library (host-side, if applicable)
```

> [!NOTE]
> The system install at `/usr/share/swordigo/` is **read-only**. On first run,
> everything is copied to `~/.local/share/swordigo-desktop/` where the user has
> full write access. The game never writes to `/usr/share/`.

## 7. Packaging Rules

- RPM/DEB packages **must bundle all runtime files**: `swordigo_boot`, `libsre.so`,
  `asset_viewer`, engine binaries, game assets, music, launcher textures.
- **No external downloads** — no Google Drive, no `curl`/`wget` in post-install scripts.
- **No nested archives** — no `tar.xz` inside the package. RPM/DEB compression
  handles this.
- Source assets/engine for packaging from `~/.local/share/swordigo-desktop/` on
  the build machine.
- Post-install script triggers the first-run copy from `/usr/share/swordigo/` →
  `~/.local/share/swordigo-desktop/`.
- Save files (`save/`) are **never** packaged.
