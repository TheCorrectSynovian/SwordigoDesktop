Technical Documentation

Deep dive into the engineering behind the Swordigo Runtime Environment (SRE).

Compatibility Stack

The SRE bridges the gap between the original ARM Android binary and the x86_64 Linux system using a multi-layered approach:

Host Layer

Native SDL2/3 platform handling, OpenGL context management, and filesystem abstraction.

Bridge Layer

JNI shims and Android system service emulators (Log, AssetManager, Configuration).

ELF Loader

Our custom ELF loader (src/loader/elf_loader.cpp) is responsible for parsing the ARM ELF32 binary, allocating memory with proper permissions, and resolving relocations.

Key Feature: Lazy Relocation

Symbol resolution is deferred until first access to minimize startup latency, utilizing a trampoline-based approach for ARM function calls.

// Example ELF Loading Snippet
ElfLoader loader("libswordigo.so");
if (!loader.load()) {
    Logger::error("Failed to map binary");
    return -1;
}
void* entry = loader.getSymbol("Java_com_touchfoo_swordigo_SwordigoLib_init");

JNI Bridge

Since the game engine expects a Java environment, SRE provides a comprehensive JNI bridge that marshals data between the ARM emulator and native C++ code.

  • Symbol Mapping: Maps Android-specific symbols to native shims.
  • Thread Management: Synchronizes the JNI environment across multiple threads.
  • String Marshalling: Converts UTF-8/UTF-16 strings between environments.

Rendering Pipeline

The original game uses GLES 1.x. SRE translates these calls to modern Desktop OpenGL 3.3+ while injecting post-processing effects into the pipeline.

FBO Scaler

Renders the game to a fixed-resolution buffer before scaling to the window size with high-quality filtering.

PostFX

Custom GLSL shaders for SSAO, bloom, and god rays that enhance the visual fidelity without altering game logic.
