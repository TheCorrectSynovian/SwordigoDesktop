How the original ARM Android binary runs on your Linux desktop — every layer explained.

System Design

Compatibility Stack

Swordigo Desktop bridges the gap between the original ARM Android binary and your x86_64 Linux system through a multi-layer compatibility stack:

🖥️
Host Platform (Linux x86_64)
SDL2 window & input · OpenGL context · OpenAL device · Native filesystem
↕ trampoline calls / marshalling
🔧
Bridge Layer
JNI bridge · Android API shims · AAssetManager → fopen · liblog → printf
↕ ARM register marshalling (R0–R3)
⚙️
Unicorn Engine (ARMv7 Emulator)
Embedded ARMv7 CPU emulator · Memory mapping · Hook points for bridge calls
↕ ELF loading · relocation · symbol resolution
📦
libswordigo.so (ARM ELF32)
Caver engine · Lua 5.1 scripting · Protobuf data · GLES 1.x renderer · OpenAL audio

ELF Loading

The Loader

Loading Process

Step 1

ELF Header Parsing

Read ELF header, verify ARM EABI5 format, iterate PT_LOAD program headers.

Step 2

Memory Mapping

Map each PT_LOAD segment into Unicorn's guest address space with correct RX/RW permissions. Zero-initialize .bss region (memsz > filesz).

Step 3

Relocation Processing

Process 10,709+ relocations: R_ARM_ABS32, R_ARM_RELATIVE, R_ARM_GLOB_DAT, R_ARM_JUMP_SLOT.

Step 4

Symbol Resolution

Internal symbols resolved to base + st_value. External imports redirected to bridge trampolines at 0xFF000000+.

Step 5

.init_array Execution

Execute all ELF initialization functions in order before calling any JNI exports.

Step 6

JNI Lifecycle

Call setFilesDir → setCacheDir → setAssetManager → setupNativeInterface → setupApplication → setApplicationViewSize → main loop.

Relocation Types

TypeDescription
R_ARM_ABS32Absolute 32-bit address
R_ARM_RELATIVEBase-relative address (most common)
R_ARM_GLOB_DATGlobal data pointer from GOT
R_ARM_JUMP_SLOTPLT jump slot for function calls

Memory Layout

Guest Memory Map:
  0x00000000 - 0x00010000: JNI env structs
  0x01000000 - 0x02800000: libswordigo.so segments
  0x02800000 - 0x0FFF0000: Guest heap / data
  0x0FFF0000 - 0x10000000: Stack (1MB)
  0xFF000000 - 0xFFFFFFFF: Bridge trampolines
✓ Critical Fix: The .bss zero-initialization bug (UC_ERR_FETCH_UNMAPPED at 0x10000000) was identified and fixed, allowing the engine to successfully reach and execute drawApplication().

Engine Startup Sequence

Android Java Path

  1. System.loadLibrary("swordigo")
  2. Native.setFilesDir(getFilesDir())
  3. Native.setCacheDir(getCacheDir())
  4. Native.setAssetManager(getAssets())
  5. GameRenderer.onSurfaceCreated()
  6. Native.setupNativeInterface()
  7. Native.setupApplication()
  8. Native.setApplicationViewSize(w, h)
  9. Loop: updateApplication() + drawApplication()

Linux Desktop Path (Our Port)

  1. Load libswordigo.so → Unicorn
  2. Build fake JavaVM + JNIEnv tables
  3. Resolve 254 external symbols → bridges
  4. Call setFilesDir (→ "./files")
  5. Call setCacheDir (→ "./cache")
  6. Call setAssetManager (→ filesystem shim)
  7. Call setupNativeInterface()
  8. Call setupApplication()
  9. SDL2 loop: updateApplication() + drawApplication()
